DFSR SYSVOL Recovery Procedure

Tutorial – Active Directory Authoritative DFSR Repair

🎯 Objective

This procedure explains how to repair a corrupted SYSVOL DFS Replication (DFSR) configuration on a Single Domain Controller environment.

The procedure covers:

  • DFSR diagnosis
  • SYSVOL recovery
  • NETLOGON restoration
  • DNS validation
  • NTP validation
  • File Server validation
  • Laptop authentication issues

🖥️ Environment

  • Single Domain Controller
  • Windows Server 2022
  • DFSR SYSVOL Replication
  • VMware Virtual Machine
  • Active Directory Domain Services

⚠️ Symptoms

  • Network drives disappearing
  • Printers requiring reconfiguration
  • Scanner shares unavailable
  • Password changes failing
  • Group Policies inconsistent
  • Authentication issues
  • Slow user logons

🧩 Root Cause

A common cause of DFSR SYSVOL failures is an incomplete Domain Controller decommissioning or migration process, leaving stale replication metadata inside Active Directory.

This left stale SYSVOL replication references inside Active Directory and caused DFS Replication to enter a protected error state.

Impact:

  • Unstable Group Policies
  • Network drive deployment failures
  • Printer deployment failures
  • Authentication inconsistencies

📋 Prerequisites

  • Administrator privileges
  • Console access to Domain Controller
  • Recent backup available
  • VMware Snapshot capability
  • PowerShell access

🚨 Important Safety Measures

  • Create a VM Snapshot
  • Backup all GPOs
  • Ensure console access
  • Perform outside business hours if possible

🔍 Verify DFSR Replication Status

Open Command Prompt on the Domain Controller.

CMD 
wmic /namespace:\\root\microsoftdfs path dfsrreplicatedfolderinfo get replicatedfoldername,state

Problematic result:

  • SYSVOL Share = 5

Healthy result:

  • SYSVOL Share = 4

📊 DFSR State Reference

State Description
0 Uninitialized
2 Initial Sync
3 Auto Recovery
4 Normal
5 Error

🔎 Run DCDIAG

Perform a complete Active Directory health assessment.

CMD 
dcdiag /v

Look for DFS Replication related errors:

  • Event ID 4012
  • Event ID 9061
  • Event ID 5002
  • Event ID 5008

These errors generally indicate stale SYSVOL data or references to an old Domain Controller.

⚙️ Review DFS Replication Events

Open Event Viewer:

GUI 
Applications and Services Logs
└ DFS Replication

Review recent events.

Common errors found in certain scenarios:

  • 4012 – SYSVOL stale
  • 9061 – Replication stopped
  • 5002 – Communication failure
  • 5008 – Partner connection failure

🔎 Verify SYSVOL and NETLOGON Shares

Verify that Active Directory shares are available.

CMD 
net share

Expected shares:

  • SYSVOL
  • NETLOGON

If one or both are missing, SYSVOL recovery is required.

⚙️ Create VMware Snapshot

Before performing any repair, create a complete VMware snapshot.

  • Snapshot Name: Before DFSR Repair
  • Include VM Memory if possible
  • Verify successful completion

This provides a rollback option if the repair fails.

⚙️ Backup All Group Policies

Open PowerShell as Administrator.

POWERSHELL 
Backup-GPO -All -Path "C:\GPOBackup"

Verify that all GPOs have been exported successfully.

🔎 Verify SYSVOL Folder Structure

Verify that SYSVOL folders exist.

CMD 
dir C:\Windows\SYSVOL

Confirm the presence of:

  • Policies
  • Scripts

⚙️ Prepare Authoritative DFSR Recovery

This procedure is intended for:

  • Single Domain Controller environments
  • State 5 DFSR failures
  • Stale SYSVOL situations
  • No remaining healthy replication partner

The following steps will rebuild DFSR and republish SYSVOL.

⚙️ Final Pre-Recovery Validation

Before continuing, confirm:

  • VM Snapshot completed
  • GPO Backup completed
  • Administrator access available
  • Console access available
  • No production changes currently running

Once validated, continue with the Authoritative DFSR Recovery procedure.

🛑 Stop DFS Replication Service

Open an elevated Command Prompt on the Domain Controller.

CMD 
net stop dfsr

Reset DFSR Local State

Open Registry Editor and browse to:

REGISTRY 
HKLM\SYSTEM\CurrentControlSet\Services\DFSR\Parameters\SysVols\Migrating SysVols\Local State

Modify:

  • Local State = 0

This resets the DFSR SYSVOL state.

▶️ Start DFS Replication Service

CMD 
net start dfsr

Wait approximately 2 to 5 minutes.

Configure Authoritative Recovery

Return to Registry Editor.

REGISTRY 
HKLM\SYSTEM\CurrentControlSet\Services\DFSR\Parameters\SysVols\Migrating SysVols

Configure:

  • Local State = 3
  • Is Primary = 1

This forces the Domain Controller to become the authoritative SYSVOL source.

Backup Existing SYSVOL_DFSR Data

Navigate to:

FILESYSTEM 
C:\Windows\SYSVOL_DFSR

Rename:

FILESYSTEM 
SYSVOL_DFSR → SYSVOL_DFSR.old

Keep the old folder as a backup.

Recreate SYSVOL_DFSR Structure

Create the following folder structure:

FILESYSTEM 
C:\Windows\SYSVOL_DFSR
└ domain

Restore Policies and Scripts

Copy the content from:

SOURCE 
C:\Windows\SYSVOL_DFSR.old\domain

To:

DESTINATION 
C:\Windows\SYSVOL_DFSR\domain

Verify that:

  • Policies folder exists
  • Scripts folder exists
  • Permissions are preserved

🔄 Restart DFSR and NETLOGON

Restart DFS Replication:

CMD 
net stop dfsr
net start dfsr

Restart NETLOGON:

CMD 
net stop netlogon
net start netlogon

Re-register Domain Controller DNS Records

CMD 
nltest /dsregdns

This forces Domain Controller DNS registration.

Validate DFSR Initialization

Open Event Viewer:

GUI 
Applications and Services Logs
└ DFS Replication

Look for:

  • Event ID 4602

Event ID 4602 confirms that SYSVOL has been initialized successfully.

🔎 Verify DFSR Status

CMD 
wmic /namespace:\\root\microsoftdfs path dfsrreplicatedfolderinfo get replicatedfoldername,state

Expected result:

  • SYSVOL Share = 4

State 4 confirms that DFSR is healthy.

Post-Recovery Network Profile Verification

In some environments, an additional issue may appear after DFSR recovery.

Once SYSVOL returned to State 4 and DFSR was healthy, several laptops were still unable to:

  • Access network drives
  • Access scanner shares
  • Change Active Directory passwords
  • Authenticate correctly against the Domain Controller

In some environments, the Domain Controller network profile may unexpectedly switch from Domain to Public after recovery operations.

Restarting the network adapter immediately restored normal operation.

POWERSHELL 
Get-NetAdapter | Restart-NetAdapter

After restarting the network adapter:

  • Network drives became accessible again
  • Scanner shares were restored
  • Password changes started working
  • User authentication returned to normal

This validation should always be performed after a DFSR recovery operation.

🔎 Verify SYSVOL and NETLOGON Shares

Confirm that SYSVOL and NETLOGON have been published successfully.

CMD 
net share

Verify that the following shares are present:

  • SYSVOL
  • NETLOGON

Missing shares indicate that SYSVOL publication is still failing.

🧪 Validate Active Directory Health

Run a complete Active Directory health assessment.

CMD 
dcdiag

Verify that:

  • Advertising = Passed
  • DFSREvent = Passed
  • SysVolCheck = Passed
  • NetLogons = Passed

🌐 Validate Active Directory DNS

Validate DNS configuration.

CMD 
dcdiag /test:dns

Expected result:

  • PASS for DNS tests

ℹ️ Known DNS Warning

In some environments, the following warning was still present:

DCDIAG 
Error:
Missing glue A record

Server:
dc01.contoso.local

This warning does not impact:

  • SYSVOL
  • NETLOGON
  • Authentication
  • Group Policies
  • DFSR replication

The environment can still be considered healthy if all other validation tests are successful.

🕒 Validate Time Synchronization

Verify Windows Time configuration.

CMD 
w32tm /query /status

Example of a healthy result:

RESULT 
Source : be.pool.ntp.org
Leap Indicator : No warning

🔐 Validate File Server Trust Relationship

Execute the following command on the File Server.

CMD 
nltest /sc_verify:CONTOSO.LOCAL

Expected result:

RESULT 
Trusted DC Name \\DC01.CONTOSO.LOCAL

Trusted DC Connection Status = 0x0

Trust Verification Status = 0x0

NERR_Success

🔃 Refresh Group Policies

Execute on the File Server.

CMD 
gpupdate /force

Expected result:

  • Computer Policy update completed successfully
  • User Policy update completed successfully

An MDM warning may appear and can generally be ignored if the server is not managed by Intune.

Validate DFSR Event Logs

Open DFS Replication logs again.

GUI 
Applications and Services Logs
└ DFS Replication

Confirm that no new critical events are generated.

The following events should no longer appear:

  • 4012
  • 5002
  • 5008
  • 9061

🏛️ Domain Controller Validation Summary

  • DFSR State = 4
  • SYSVOL Share Published
  • NETLOGON Share Published
  • DCDIAG Successful
  • DNS Operational
  • NTP Operational
  • No DFSR Critical Events

If all items above are validated, the Domain Controller can be considered healthy.

Workstation Validation

After DFSR recovery and Domain Controller network adapter restart, most workstations should automatically recover.

If some users still experience issues, perform the following checks on the affected workstation.

Common symptoms:

  • Network drives unavailable
  • Scanner shares inaccessible
  • Password change failures
  • Authentication issues
  • Missing printers

Step 1 - Verify Logon Server

CMD 
echo %logonserver%

Expected result: 

\\DC01

Step 2 - Verify Domain Controller Discovery

CMD 
nltest /dsgetdc:CONTOSO.LOCAL

Step 3 - Verify Network Configuration

CMD 
ipconfig /all

Verify that the workstation is using the Active Directory DNS server.

Step 4 - Disable IPv6 (If Required)

If IPv6 DNS addresses are detected and authentication problems persist, disable IPv6 on the workstation network adapter.

Step 5 - Flush DNS Cache

CMD 
ipconfig /flushdns
ipconfig /release
ipconfig /renew

Step 6 - Refresh Group Policies

CMD 
gpupdate /force

Step 7 - Validate User Access

  • Password changes working
  • Network drives accessible
  • Printers available
  • Scanner shares accessible
  • User authentication successful

☑️ Final Validation Checklist

  • ✅ DFSR State = 4
  • ✅ SYSVOL Share Published
  • ✅ NETLOGON Share Published
  • ✅ DFSR Event ID 4602 Present
  • ✅ No DFSR Errors 4012
  • ✅ No DFSR Errors 5002
  • ✅ No DFSR Errors 5008
  • ✅ DCDIAG Successful
  • ✅ DNS Operational
  • ✅ NTP Operational
  • ✅ File Server Trust Verified
  • ✅ GPUpdate Successful
  • ✅ Password Changes Working
  • ✅ Scanner Accessible
  • ✅ Network Drives Accessible
  • ✅ User Authentication Operational

🎉 Conclusion

A common cause of DFSR SYSVOL failures is an incomplete Active Directory migration or Domain Controller decommissioning process.

This resulted in stale SYSVOL replication information and caused DFS Replication to enter a protected error state.

An authoritative DFSR recovery was performed, SYSVOL was rebuilt, NETLOGON was republished and Active Directory functionality was restored.

In some environments, workstation authentication issues may be related to DNS or IPv6 configuration and should be investigated as part of the post-recovery validation process.

In some environments, the Domain Controller network profile may switch from Domain to Public after recovery operations. If this occurs, restarting the network adapter may restore normal connectivity.

The environment was validated successfully and all critical services were restored.
